# Agent Spec — cs-ticket-agents

Sistema multiagente para automatizar el diagnóstico de tickets de customer support de Runa (proyecto `saas-rails-api`, nómina México).

## Overview

Los tickets de CS llegan a un grupo de Gmail. Hoy Julián (CS) los resuelve manualmente: diagnostica en la consola Rails de producción y ejecuta scripts one-liner. Este sistema automatiza el diagnóstico:

1. Un **listener** recibe la notificación de mail nuevo (Gmail `users.watch` → Pub/Sub → pull local).
2. Un **orquestador** (root_agent) clasifica el ticket por tipo (IDSE, nómina, variabilidad, otro) y delega al subagente especializado.
3. El **subagente** consulta la base de runbooks (vía servidor MCP propio), analiza los adjuntos (excel/imágenes), pide datos de consola al humano si los necesita, y entrega: **diagnóstico + script one-liner propuesto + runbook usado**.
4. Julián ejecuta el script manualmente en producción. La resolución se registra en el índice (`log_resolution`) para enriquecer la base de conocimiento.

**v1 es solo diagnóstico (read-only).** El agente NUNCA ejecuta nada en producción. La ejecución automática con gate de aprobación queda para v2, pero las interfaces de las tools se diseñan para que el cambio sea solo de backend.

### Versatilidad de modelos (requisito explícito)

El modelo de cada agente es **configuración, no código**: string por env/config (ej. `gemini-flash-latest`, `anthropic/claude-haiku-4-5`, `openai/gpt-...`). Gemini nativo como default; modelos no-Gemini vía `LiteLlm`. Orquestador y subagentes pueden usar modelos distintos.

## Example Use Cases

1. **IDSE pendiente con lote enviado**: ticket "los movimientos afiliatorios de la empresa X figuran pendientes pero el lote ya se envió". El agente busca el runbook `idse-pendientes-lote-enviado`, pide a Julián los IDs de movements/request por consola, diagnostica que hay que vincular movements al request del lote y re-correr `SaveDetailedResponse`, y propone los one-liners exactos.
2. **Vacaciones infladas tras reingreso**: ticket con la proporción de vacaciones incorrecta. El agente identifica el patrón (balance residual del ciclo anterior), pide el detalle de `request_balances` por consola y propone el script de corrección + recálculo.
3. **Ticket con excel adjunto**: cliente manda un xlsx con empleados afectados. El agente lo parsea, extrae los IDs y los usa en el diagnóstico.
4. **Ticket no clasificable**: el orquestador lo etiqueta "otro", no inventa diagnóstico, y lo deja para atención manual con un resumen.

## Tools Required

| Tool | Propósito | Auth / detalle |
|---|---|---|
| `search_runbooks`, `get_runbook`, `log_resolution`, `find_similar_tickets` | Base de conocimiento (repo git de runbooks md + índice de resoluciones) | Servidor MCP propio (FastMCP), consumido con `MCPToolset` |
| `read_excel` | Parsear adjuntos xlsx/csv a JSON | pandas/openpyxl, local |
| Lectura de imágenes | Screenshots en tickets | Nativa del modelo (bloques de imagen); solo pipeline de adjuntos |
| `request_console_data` | Pedir al humano el output de una consulta read-only en la consola Rails de producción | v1: human-in-the-loop (el agente entrega el one-liner, Julián pega el output). Interfaz: entrada `ruby_statement` (read-only), salida `console_output`. v2: mismo contrato, backend AWS SSM contra la instancia EC2 (`runa-access` → `connect_prod_api_bash` → `rails c`) |
| Gmail API | Leer mails del grupo + adjuntos; `users.watch` | OAuth de la cuenta miembro del grupo; scope readonly en v1 |
| Pub/Sub | Notificaciones de mail nuevo | Suscripción **pull** (sin endpoint público, funciona local) |

## Constraints & Safety Rules

- **Prohibido ejecutar escritura en producción.** v1 no tiene ningún camino de ejecución: toda consulta de consola pasa por el humano.
- Los statements que el agente proponga para consola deben ser **one-liners** (cada sentencia pegable en una sola línea, preferencia de Julián) y estar separados en: (a) consultas de diagnóstico read-only, (b) script de corrección propuesto — claramente marcado como "para revisión humana".
- Si el agente no encuentra runbook aplicable ni patrón conocido, debe decirlo y escalar — **nunca inventar un diagnóstico**.
- Datos de nómina/PII: en desarrollo se usan tickets anonimizados (tier gratis de Gemini API puede usar datos para entrenamiento). Con datos reales → API paga o Vertex AI.
- Idioma: toda la interacción con Julián en español.

## Success Criteria

- Con los 2 casos core del eval (IDSE + nómina, derivados de sesiones reales anonimizadas): clasificación correcta del tipo de ticket, runbook correcto recuperado, diagnóstico correcto y script propuesto equivalente al que se usó realmente.
- Trayectoria de tools correcta (consulta runbooks antes de diagnosticar; pide datos de consola en vez de inventarlos).
- La misma eval pasa con al menos dos modelos distintos (Gemini y Claude vía LiteLlm) sin cambios de código (`eval compare`).
- Ticket no clasificable → escalado, no alucinado.

## Architecture & Sub-Agents

- `root_agent` (orquestador/clasificador): recibe el ticket normalizado (asunto, cuerpo, adjuntos parseados), clasifica y delega vía `sub_agents`.
- `agente_idse`: movimientos afiliatorios, lotes IDSE, fechas IMSS.
- `agente_nomina`: cálculos de nómina, percepciones, vacaciones, recurrencias.
- Extensible: agregar un subagente = nueva instrucción + registro en el orquestador (los runbooks viven en el MCP, no en el prompt).
- Listener de Gmail: proceso separado del sistema de agentes (mismo repo, entrypoint propio) para poder migrar pull→push al deployar sin tocar agentes.
